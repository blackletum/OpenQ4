# openQ4 v0.2.000

## Highlights

- Fixes the initial Windows `v0.2.000` package startup failure where the client could show the splash screen and exit before reaching the menu.
- Makes macOS package launches more reliable from Finder and Terminal by resolving the real executable location, validating the packaged `baseoq4/` directory before startup, and embedding the packaged client as the `.app` executable.
- Adds macOS ARM64 OpenGL and Metal bridge build/install coverage to pull-request and push validation, and tightens macOS package checks before release archives are produced.
- Improves Linux SDL3 Wayland handling so logs include active Wayland hints, display content scale, orientation, current/desktop display modes, exact refresh details when SDL reports them, and compositor-accepted window state after screen changes; direct client launches can use `OPENQ4_FORCE_X11=1` to force XWayland when native Wayland has driver or compositor issues; `OPENQ4_WAYLAND_PREFER_LIBDECOR=1` can opt into SDL's libdecor preference for compositor decoration issues; `OPENQ4_WAYLAND_SYNC_WINDOW_OPS=1` can opt into SDL's blocking window-operation synchronization for diagnosis; native Wayland mouse-look can continue when pointer confinement is unavailable; and fullscreen/windowed transitions refresh after SDL synchronizes compositor-owned window state.
- Improves SDL3 Linux VRAM autodetection in native Wayland and minimal Linux sessions by enumerating DRM card/render-node sysfs VRAM totals before falling back to known nodes and legacy `/proc/dri` probing.
- Makes early Linux Wayland startup messages point to openQ4's XWayland, libdecor, and window-sync troubleshooting switches before SDL video initialization.
- Improves SDL3 Linux desktop-resolution detection by falling back to current display mode and display bounds when a compositor does not report a conventional desktop mode.
- Keeps the generated macOS app metadata aligned with the supported macOS 11.0 package floor.
- Improves macOS package diagnostics when the embedded app executable drifts from the adjacent packaged client binary.
- Rejects macOS release packages that accidentally link against unbundled Homebrew/local/runtime-loader dependencies before they can become launch-time failures for players.
- Documents the supported `openQ4.app` launch path on macOS and the Gatekeeper quarantine workaround for unsigned archive builds.
- Physical/DVD-style Quake 4 installs that include the core retail media set now pass startup validation without needing Steam-added patch/menu PK4s such as `pak023.pk4` through `pak025.pk4`.
- The packaged `baseoq4` manifest is now generated from the actual engine release version, so release archives and installers stay loadable when the workflow bumps versions.
- `requiredopenQ4Version` is treated as a minimum compatible engine version, while the manifest's own `version` remains the packaged content version.
- Stable release packaging now fails early if `baseoq4/mod.json` has a stale content version or requires a newer engine than the package contains.
- Strogg health stations can be used again after the player becomes Strogg in the single-player campaign.

## Upgrade Notes

- Replace any earlier `v0.2.000` Windows package or installer that closes immediately after the splash screen.
- You do not need to fetch optional Steam patch/menu PK4s if your legitimate Quake 4 install already has `pak001.pk4` through `pak022.pk4`.
- No save-game, config, or asset changes are required.
- On Linux or Steam Deck, set `OPENQ4_FORCE_X11=1` only if you need the XWayland fallback; use `OPENQ4_WAYLAND_PREFER_LIBDECOR=1` only for native Wayland decoration or resize issues; use `OPENQ4_WAYLAND_SYNC_WINDOW_OPS=1` only for diagnosis because compositor animations may block; explicit `SDL_VIDEO_DRIVER` or `SDL_VIDEODRIVER` settings still take precedence.

## Change Log

- Hardened macOS SDL3 launch initialization through `SDL_RunApp()` and improved default base-path discovery for packaged app launches.
- Added a direct macOS app-parent base-path probe so launches from `openQ4.app/Contents/MacOS` can find the extracted package root even before Cocoa bundle path fallbacks are needed.
- Honored `OPENQ4_FORCE_X11=1` directly in the SDL3 Linux backend when no explicit SDL video-driver override is set, added `OPENQ4_WAYLAND_PREFER_LIBDECOR=1` as an opt-in SDL libdecor preference, added `OPENQ4_WAYLAND_SYNC_WINDOW_OPS=1` as an opt-in SDL window-operation synchronization diagnostic, logged active native Wayland hint values plus display/window scale, orientation, and mode diagnostics, expanded `listDisplayModes` with SDL scale, pixel-density, and exact-refresh details, made native Wayland pointer confinement best-effort after relative mouse mode succeeds, and synchronized SDL window state before refreshing renderer placement after screen mode changes.
- Added enumerated DRM card/render-node sysfs VRAM probing to the SDL3 Linux backend so X11-free sessions have a modern non-XNVCtrl autodetect path before falling back to conservative defaults.
- Updated early Linux Wayland runtime reporting to include `OPENQ4_FORCE_X11`, `OPENQ4_WAYLAND_PREFER_LIBDECOR`, and `OPENQ4_WAYLAND_SYNC_WINDOW_OPS`.
- Added SDL3 Linux desktop-resolution fallbacks from desktop mode to current mode and display bounds.
- Updated the macOS `.app` bundle so Finder launches an embedded copy of the packaged client binary.
- Expanded macOS pull-request and push validation to build/install both OpenGL and Metal bridge SDL3 variants.
- Added macOS staged-payload and generated `.app` validation, including executable, icon, splash, and `Info.plist` checks.
- Updated macOS app metadata so `LSMinimumSystemVersion` matches the Meson deployment target.
- Added long-path handling for macOS executable discovery, release checks that compare the embedded app executable with the packaged client binary, and stricter checks for key app plist values.
- Added `otool -L` dependency validation for macOS app, client, dedicated-server, and game-module binaries so release packages stay free of unbundled non-system dynamic-library links.
- Added final macOS release archive validation for required app bundle entries, executable bits, localized plist resources, safe paths, and app plist identity/version values.
- Made the Bash build and validation wrappers used by macOS CI/release packaging bootstrap without GNU-style `dirname --`, keeping them compatible with the stock macOS command-line userland.
- Aligned the legacy macOS `Info.plist` template with packaged app identity metadata and added a parsed plist regression check so native/Xcode fallback metadata stays in sync.
- Tightened macOS staged-payload validation so release builds require matching no-extension engine binaries, `.dylib` game modules, and no stale Linux/Windows game module artifacts before packaging.
- Corrected release staging so `baseoq4/mod.json` uses the effective Meson `version_base_override` value.
- Limited required official media validation to the core retail PK4 baseline and documented optional official patch/menu and language packs.
- Added a packaging validation guard for stable releases to catch stale manifest versions and impossible `requiredopenQ4Version` requirements before artifacts are published.
- Allowed animated in-world GUI models to participate in GUI tracing, restoring the stock health-station `heal begin` / `heal end` interaction path.
