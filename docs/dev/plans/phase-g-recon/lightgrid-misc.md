## 2. Light-grid rendering is inert under the vk pins — confirmed

- `VK_FillGLConfigFromDevice` (`src/renderer/Vulkan/vk_Backend.cpp:62-110`) pins NPOT/S3TC/BPTC/cubeMap/allowARB2Path/texture-unit counts only. `glConfig.GLSLProgramAvailable` is **never set** under vk — the only assignment is in the GL extension probe (`RenderSystem_init.cpp:1141`, inside R_CheckPortableExtensions), which VK_InitRenderDevice replaces; it stays false from zero-init. Matches the F recon record (`docs/dev/plans/phase-f-recon/light-frontend.md:56`).
- ScenePackets: `R_ScenePackets_DrawSurfLightGridEligible` (`src/renderer/ScenePackets.cpp:1273-1301`) rejects every surface at :1274 on `!glConfig.GLSLProgramAvailable` → the RENDER_PASS_LIGHT_GRID packet pass (:1656) is always empty under vk. Additionally scene-packet front-end capture only runs at all when `r_rendererMetrics >= 2` (:48-50) — packets are a metrics/side pipeline, not the vk render path.
- The GL draw pass itself (`RB_STD_LightGridIndirect` et al.) lives in draw_common.cpp, which is not even compiled into the vk module.
- Net: **under vk nothing samples, streams, or draws light-grid data**. Consequence for Phase G: light-grid parity on vk cannot be reached by flipping a gate — it requires a NEW vk pass (SPIR-V port of lightgrid_indirect.fs semantics + atlas materialization through vk_Image; `EnsureLightGridAreaImages` already works mechanically under vk since AllocImage/SubImageUpload route through the vk image backend). If G defers it, G is stencil+fog/blend only, and that is behavior-compatible with GL-at-defaults minus the (default-on) light-grid overlay — a visible parity gap on maps shipping .lightgrid files (content/baseoq4/pak1/maps/** ship them broadly).
- Perf note (minor, speculation as to whether it matters): under vk with grids loaded, `AnyLightGridAvailable()` is true so the per-drawSurf PointInArea BSP descent (`tr_light.cpp:2024`, gate `tr_main.cpp:1278-1280`) runs every frame with zero consumers. A vk-side pin (skip area resolve until a vk light-grid pass exists) is free perf.

## 4. RB_ShadowMapResourcesKnownGood / F3 vs stencil coexistence

- vk stub today: `RB_ShadowMapResourcesKnownGood` returns **false unconditionally** (`src/renderer/Vulkan/vk_Backend.cpp:792-795`) → the front-end never elides stencil volumes under vk; F1 lights simply draw unshadowed.
- The elision gate `R_ShadowMapLightWillUseShadowMaps` (`src/renderer/tr_light.cpp:1419-1448`) checks **in order**: `r_shadowMapSkipStencilShadows` (default 1, `RenderSystem_init.cpp:321`) AND `r_useShadowMap` (default **0**, :288) AND `r_shadows` (:1420); `r_shadowMapMaxUpdatesPerView > 0` disables elision wholesale (:1424); per-light sticky `lightDef->shadowMapStencilFallbackSticky` (:1427); light/caps checks; **only then** `RB_ShadowMapResourcesKnownGood(pointLightPath)` (:1447). Consumers of the verdict: prelight volumes (`tr_light.cpp:1464`), dynamic interaction shadow-tri linking (`src/renderer/Interaction.cpp:2108`, also :1629), GL backend (`draw_arb2.cpp:10877` — excluded under vk).
- **Coexistence contract is safe by gate order**: with r_useShadowMap=0 (the default the G stencil path serves), line :1420 short-circuits before resource-known-good is ever consulted — flipping the vk stub true (F3) cannot shed volumes on the default path. Nothing breaks for r_useShadowMap 0.
- What F3 actually changes: with r_useShadowMap=1, front-end elision activates per light class. The GL reference implements per-class **video-generation** honesty: `g_shadowMapProjectedResourcesOkGeneration` / `g_shadowMapPointResourcesOkGeneration` vs `tr.videoRestartCount` (`draw_arb2.cpp:2883-2892`) — "the front end must keep generating stencil volumes until the backend has proven it can create shadow-map resources in this video generation." The failure contract is `RB_ShadowMapMarkStencilFallbackSticky` (`draw_arb2.cpp:2942-2951`): a shadow-map render failure on a volume-elided light sets the sticky flag so volumes regenerate **from the next frame**; the failing frame renders without that light's shadow.
- **The real G dependency**: after F3, a sticky-fallback light needs the stencil path to exist to draw its restored volumes — under F1/F2-only vk, fallback lights stay unshadowed forever (the F plan encodes this: "Failure → draw unshadowed (no stencil until G)", `docs/dev/plans/2026-07-19-vulkan-phase-f.md:63-67`). Conversely once G lands, the vk interaction pass must handle BOTH per-light modes in one view (stencil-tested receivers for volume lights, ALWAYS-128 receivers for mapped lights — GL does this at draw_arb2.cpp:11546 vs :11643), and the vk RB_ShadowMapResourcesKnownGood must adopt the per-class generation pattern (reset on swapchain/device recreate, the vk analog of videoRestartCount) rather than a boolean flip, or a mid-session device reset would let the front-end elide volumes the backend can no longer service.
