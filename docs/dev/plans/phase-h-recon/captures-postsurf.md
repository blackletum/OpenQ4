**How _currentRender is consumed:**
- Intrinsics: `_currentRender` = RGBA16F, `_currentDepth` = FMT_DEPTH (D32F under vk), created at `src/renderer/Image_intrinsic.cpp:703-704` (256×256 DEFAULT_SIZE placeholders until first copy resizes them); `originalCurrentRenderImage` saved at 718 for stereo.
- Material promotion: any stage referencing `_currentRender*`/`_currentDepth*` (or the intrinsic pointers) gets `sort = SS_POST_PROCESS` + `MC_TRANSLUCENT` **unless** the material says `needCurrentRender` (`MF_NEED_CURRENT_RENDER`, Material.cpp:2723-2725), which preserves authored sort — `src/renderer/Material.cpp:3161-3199`, `R_IsSceneCaptureImage` at Material.cpp:214-232 (name-prefix match `_currentRender`/`_currentDepth`).
- Back-end walk: `RB_STD_DrawShaderPasses` (`src/renderer/draw_common.cpp:7046-7175`). If the list starts at `sort >= SS_POST_PROCESS` it captures `_currentRender` (and `_currentDepth` if any material needs it, 7087-7106) via `RB_CaptureCurrentRenderImage`/`RB_CaptureCurrentDepthImage` (draw_common.cpp:1512-1554 — viewport-origin, viewport-sized copies; skip-as-satisfied when rendering into a target whose color/depth image *is* the capture image). Mid-walk: `MF_NEED_CURRENT_RENDER` + authored sort < SS_POST_PROCESS triggers a capture just before that surface draws (7146-7156). **The break**: draw_common.cpp:7158-7162 — surfaces with `sort >= SS_POST_PROCESS` stop the walk if `!backEnd.currentRenderCopied`; only capture-allowed contexts (`RB_AutomaticCurrentRenderCaptureAllowed`, 1489-1493: no render texture, or scene/feedback texture) set the flag. Per-stage lazy capture also exists inside `RB_STD_T_RenderShaderPasses` (6593-6598).
- Flags reset per view in `RB_BeginDrawingView` (`src/renderer/tr_render.cpp:1049-1050`), and re-cleared after fog (draw_common.cpp:9807-9808) and light-grid inline submits (7167) to force re-capture.
- Shader scale plumbing: `RB_SetProgramEnvironment` (draw_common.cpp:6367-6434) sets ARB env[0] = `(viewportW/texW, viewportH/texH)` from `currentRenderImage->GetOpts()` ("POT correction" — identity 1.0 now that NPOT storage matches viewport) and env[1] = `1/w, 1/h` window-to-UV. GLSL equivalents: `currentRenderViewportOrigin/Size`, `currentRenderTextureScale` shader parms (draw_common.cpp:545-569; parsed at Material.cpp:1514-1522).

## 2. Post-process surface pass + texgen math + _currentDepth users

- Frame order in `RB_STD_DrawView` tail: `processed = RB_STD_FindPostProcessStart` (first surf with sort ≥ SS_POST_PROCESS, draw_common.cpp:7364-7371); pre-fog walk (9769-9775), fog (9806), post-fog walk (9817-9819), SSAO/MotionBlur/Bloom (9824-9830), then **the post-process surface pass**: `RB_STD_DrawShaderPasses(drawSurfs+processed, numDrawSurfs-processed)` at draw_common.cpp:9833-9835 — this is what the vk backend currently cuts off (vk walk stops at the same boundary: vk_GuiExecutor.cpp:2244-2253).
- **Texgen keywords** (Material.cpp:1913-1961): `mirrorRenderMap`/`refractionRenderMap`/`xrayRenderMap` and bare `screen` → TG_SCREEN; `screen2` → TG_SCREEN2; `glassWarp` → TG_GLASSWARP. Retail q4 users: glass/refraction distortion (glassWarp + `_currentRender` via scratchImage), heat-haze ARB materials sampling `_currentRender` (no assets in repo — retail paks; speculation grounded in `FPROG_GLASSWARP` programs draw_arb2.cpp:11703-11704 and MF_NEED_CURRENT_RENDER parsing).
- **TG_SCREEN/TG_SCREEN2 math** (`RB_PrepareStageTexturing`, draw_common.cpp:4436-4488; identical for both): fixed-function `GL_OBJECT_PLANE` texgen with S/T/Q rows = rows 0,1,3 of `modelView*projection` (`myGlMultMatrix`), i.e. tc = clip-space xyw; the projective divide (s/q, t/q) yields NDC [-1,1]→ the sampled texture's [0,1] via the ARB program's env[0] scale. GL convention: t=0 at framebuffer bottom, matching CopyFramebuffer's bottom-left-origin copy. Under vk the executor already flips via negative-height viewport (vk_GuiExecutor.cpp:2007-2020), so a swapchain→_currentRender blit that keeps GL row order needs a Y-flipped blit region (or flip in shader) — the copied image must look bottom-up to the TG_SCREEN math for parity.
- **_currentDepth users**: soft particles — `RB_TryDrawSoftParticleStage` (draw_common.cpp:4849-4975) lazily calls `RB_CaptureCurrentDepthImage` (4865-4867), binds `currentDepthImage` on unit 1 as `SceneDepth` with uniforms depthProjection (proj[10], proj[14]), viewportOrigin, invDepthTexSize (4921-4932); gate `RB_DrawSurfHasSoftParticleStage` (4802+, `r_softParticles` + GLSL). Also GLSL stages via `RB_StageUsesCurrentDepth`/`RB_MaterialUsesCurrentDepth` (draw_common.cpp:110-145), motion blur (2793), SSAO final depth (2098-2122), and RV special-effects depth passes. Note the vk module stubs `RB_DrawSurfHasSoftParticleStage` to false (vk_Backend.cpp:516-519).

## 3. What vk needs (current state of the vk module)

- **vk source set**: `tools/build/meson_sources.py:112-126` — vk module compiles ALL renderer TUs except the listed GL backend ones. **`Image_load.cpp` and `Image_intrinsic.cpp` ARE compiled in**, so `idImage::CopyFramebuffer/CopyDepthbuffer` exist under vk but execute GL-1.1 no-op stubs (`vk_GLStubs.cpp:277-278`; GLEW is the never-initialized "dedicated" flavor, vk_GLStubs.cpp:14-17,378-379 — all `GLEW_*` capability checks are false so the non-blit branches run harmlessly). They're currently unreachable anyway: `RB_ExecuteBackEndCommands` in `src/renderer/Vulkan/vk_Backend.cpp:336-387` handles only RC_NOP/RC_SET_BUFFER/RC_DRAW_VIEW/RC_SWAP_BUFFERS; RC_COPY_RENDER and all RC_SET_* fall into `default:` (382-384, "copy/capture commands arrive with Phase E+"). CAUTION: if RC_COPY_RENDER is serviced by calling the compiled `CopyFramebuffer`, it will silently mutate `opts.width/height/internalFormat` while copying nothing — a vk implementation should override at the RB_CopyRender-equivalent level or add vk-specific idImage copy methods in vk_Image.cpp.
- **idRenderTexture under vk**: bookkeeping-only, `MakeCurrent` returns false (vk_Backend.cpp:434-441), so `RB_AutomaticCurrentRenderCaptureAllowed` logic degenerates to backEnd.renderTexture==NULL (RB_SetRenderTexture also lives in excluded tr_backend.cpp).
- **texnum→VkImage table**: `src/renderer/Vulkan/vk_Image.h:21-40` — `vkImageEntry_t {image, allocation, view, sampler, format, w, h, generation}` in a 4096-slot table; `idImage::texnum` is the slot index (`GetDeviceHandle()` returns texnum, Image.h:173). `AllocImage` (vk_Image.cpp:348-431) creates usage `SAMPLED|TRANSFER_DST` — already sufficient as a blit/copy **destination**. FMT_RGBA16F→`VK_FORMAT_R16G16B16A16_SFLOAT`, FMT_DEPTH→`VK_FORMAT_D32_SFLOAT` (vk_Image.cpp:156-163). Descriptor binding is generation-keyed: `VK_GuiExecutor_GetImageDescriptor` (vk_GuiExecutor.cpp:711-753) re-writes the cached set when `entry->generation` changes — so a capture implementation should bump `generation` (and set `everUploaded`) after the first copy so stale descriptors refresh; subsequent per-frame copies into the same VkImage need no descriptor churn.
- **Copy source problem**: swapchain images are created `COLOR_ATTACHMENT|TRANSFER_DST` only (`src/renderer/Vulkan/VulkanDevice.cpp:279`) — **`VK_IMAGE_USAGE_TRANSFER_SRC_BIT` must be added** for both _currentRender capture and screenshot readback. Swapchain format: prefers B8G8R8A8_UNORM/R8G8B8A8_UNORM SRGB_NONLINEAR (VulkanDevice.cpp:202-209). Since _currentRender is RGBA16F, `vkCmdCopyImage` is illegal (format size mismatch) — use `vkCmdBlitImage` (does format conversion + the Y-flip via inverted blit offsets), or allocate a swapchain-format capture image and swap the intrinsic's entry. Per-slot depth images (`vkCtx.depthImages`, probed D24S8/D32S8, VulkanDevice.cpp:118-128) are `DEPTH_STENCIL_ATTACHMENT` usage only (VulkanDevice.cpp:147) — need TRANSFER_SRC for _currentDepth (note vkCmdBlitImage cannot convert depth formats; D24S8→D32F requires a depth-sampling fullscreen pass or matching the _currentDepth VkFormat to vkCtx.depthFormat with a depth-only view).
- **Where the copy goes in the frame**: copies/blits must happen outside the dynamic-rendering scope. The Phase F2a suspend/resume seam is exactly the pattern: `VK_Exec_EndMainRendering` → work → `VK_Exec_BeginMainRendering(false)` resumes with loadOp LOAD for color+depth (vk_GuiExecutor.cpp:1277-1351). Two constraints: (a) depth storeOp is `DONT_CARE` unless shadow maps are on (1315-1316) — must be STORE whenever a depth capture may follow the scope break; (b) swapchain image is in COLOR_ATTACHMENT_OPTIMAL inside the frame — transition to TRANSFER_SRC_OPTIMAL, blit, transition back before resume.
- **Scale factors/coords**: with NPOT storage the GL path keeps texture size == viewport size, so env[0]≈(1,1) (draw_common.cpp:6398-6410); a vk _currentRender sized to viewport preserves the same contract. Viewport/scissor Y-flip conventions already established at vk_GuiExecutor.cpp:1489-1512 (scissor: `offset.y = fbHeight - scYGL - scH`) and 2007-2020.
- The vk 3D walk already computes `processed` and stops at SS_POST_PROCESS with a Phase-H comment (vk_GuiExecutor.cpp:2244-2253); the post-process surface pass slots in after fog/blend (Phase G) at that point.

## 4. Screenshots / readback

- `idRenderSystemLocal::TakeScreenshot` — `src/renderer/RenderSystem_init.cpp:2320-2390`: sets `tr.takingScreenshot`, `R_ReadTiledPixels` (2224-2306) tiles window-size chunks; per tile: `session->UpdateScreen()` (or direct RenderScene for `ref`), `glReadBuffer(GL_BACK)`, `glReadPixels(0,0,w,h,GL_RGB)` (2284). GL keeps the frame readable because `RB_SwapBuffers` suppresses the actual present while `tr.takingScreenshot` (tr_backend.cpp:560-564). TGA header is built inline (RenderSystem_init.cpp:2363-2377, bottom-up rows as glReadPixels emits, RGB→BGR swap) and written via `fileSystem->WriteFile`. Console cmd `screenshot` → `R_ScreenShot_f` (2916-2970). `CaptureRenderToFile` (RenderSystem.cpp:1722-1754) is the other readback: crop-region glReadPixels + `R_WriteTGA` (imagetools).
- Under vk: `glReadPixels` is a no-op stub (vk_GLStubs.cpp:278) → the R_StaticAlloc buffer stays uninitialized = garbage TGAs. Also vk's `GLimp_SwapBuffers` (vk_Backend.cpp:247-276) presents unconditionally — no takingScreenshot suppression.
- **vk readback shape**: arm a flag (takingScreenshot / explicit request); in `VK_GuiExecutor_EndFrameAndPresent` (vk_GuiExecutor.cpp:1353+) after `VK_Exec_EndMainRendering` and **before** the PRESENT_SRC barrier (1363-1379), transition swapchain image to TRANSFER_SRC_OPTIMAL, `vkCmdCopyImageToBuffer` into a host-visible VMA buffer, then transition to PRESENT_SRC; wait the frame's fence/timeline before mapping (frames-in-flight: `VK_FRAMES_IN_FLIGHT`, VulkanDevice.h:88); convert BGRA→RGB rows bottom-up to feed the existing TGA writer. Needs swapchain TRANSFER_SRC usage (VulkanDevice.cpp:279).
- **Deterministic capture gate**: `tools/tests/renderer_gameplay_benchmark.py` — issues console `screenshot "screenshots/renderer-bench/{role}_{n}.tga"` (line 548,588), locates the TGA under savepath (624-646), sha256-compares against reference dirs (807-856). Phase B used sha256-identical TGAs gl-static vs gl-module (docs/dev/plans/2026-07-16-vulkan-renderer-phase-b.md:162). So the vk readback feeds the exact same `TakeScreenshot` file path — no new tooling needed once glReadPixels-equivalent data lands in the buffer.
